# Implementation

This document provides a detailed explanation of Sonexa's internal implementation.

Unlike the architecture documentation, which describes the organization of the system, this document focuses on the runtime behavior of the application and the execution flow behind its core components.

Every major subsystem—including playback, media discovery, state management, database synchronization, and hardware audio processing—is explained from the moment a user interacts with the interface until audio reaches the device's speakers.

---

Audio playback is the central workflow of Sonexa.

# AudioController Workflow

The AudioController serves as the operational bridge between the Presentation layer and the playback engine.

Rather than exposing ExoPlayer directly to ViewModels, the controller receives playback requests, prepares media items, communicates with the MediaBrowser, and publishes playback state back to the application.

Typical responsibilities include:

- Loading media
- Queue management
- Playback commands
- Playback state propagation
- Current song updates
- Playback position updates
- Audio Session ID distribution

The controller therefore becomes the application's central playback coordinator.

# MediaItem Creation

Before playback begins, every Song object is transformed into a Media3 MediaItem.

This conversion encapsulates all metadata required by the playback engine, including:

- Media URI
- Song title
- Artist name
- Album information
- Artwork
- Playback metadata

MediaItem acts as the standard communication format used throughout Media3.

Using this abstraction allows Sonexa to support both local and remote media sources without modifying the playback engine.

---

# ExoPlayer Initialization

The ExoPlayer instance is created during the initialization of the MediaSessionService.

During service startup the following components are initialized:

- ExoPlayer
- MediaSession
- Player.Listener
- Notification Provider
- Audio Attributes

This initialization occurs only once.

Every screen within the application communicates with the same player instance throughout its lifetime.

Maintaining a single player instance reduces memory consumption while ensuring uninterrupted playback.

---

# Player State Monitoring

A Player.Listener continuously observes playback events generated by ExoPlayer.

Examples include:

- Idle
- Buffering
- Ready
- Playing
- Paused
- Ended

Whenever the playback state changes, the listener immediately forwards updated information to the AudioController.

The AudioController then updates the corresponding StateFlow objects observed throughout the application.

This event-driven approach removes the need for polling while ensuring that playback information remains synchronized across every interface.

# Playback Position Updates

Playback position is updated independently from the player state.

A coroutine periodically retrieves the current playback position from ExoPlayer at a fixed interval.

The updated value is emitted through StateFlow, allowing playback progress indicators to remain synchronized without blocking the main thread.

This implementation provides smooth progress updates while maintaining low CPU utilization.

# Media Discovery

The music discovery process begins when Sonexa scans the device's media library using Android's MediaStore API.

Rather than performing recursive file system traversal, the application queries Android's indexed media database through the ContentResolver. This approach is significantly faster, fully compatible with Scoped Storage, and avoids unnecessary filesystem operations.

The scanning process retrieves metadata including:

- Song Title
- Artist
- Album
- Duration
- Album Artwork
- Media URI

The retrieved metadata is transformed into Sonexa's unified `Song` model before entering the repository layer.

# Playlist Management

Playlists are managed independently from playback.

The repository coordinates all playlist-related operations while Room maintains persistent storage.

Supported operations include:

Songs themselves are never duplicated.

# AudioFX Implementation

One of Sonexa's most distinctive implementation details is its direct integration with Android's native AudioFX framework.

Rather than processing audio entirely in software, Sonexa attaches audio effects directly to ExoPlayer's active Audio Session ID, allowing compatible devices to utilize hardware Digital Signal Processing (DSP).

The initialization sequence follows the workflow below.

```text
Playback Starts

        │

        ▼

ExoPlayer

        │

        ▼

Audio Session ID

        │

        ▼

AudioController

        │

 ┌──────┼─────────────┬──────────────┐

 ▼      ▼             ▼

Equalizer

Bass Boost

Virtualizer

        │

        ▼

Android Audio Hardware
```

Each AudioEffect instance remains attached to the active playback session until Android generates a new Audio Session ID.

When playback switches to another output device—such as Bluetooth headphones—the AudioController automatically retrieves the updated session identifier and reattaches the configured effects.

This ensures that user-defined audio enhancements remain active throughout the listening experience.

# Audio Focus Handling

Sonexa fully participates in Android's Audio Focus system.

Whenever another application requests temporary or permanent audio focus, Media3 automatically dispatches focus events to the playback engine.

Typical scenarios include:

- Incoming phone calls
- Voice navigation instructions
- Alarm playback
- Competing music applications

Depending on the received focus event, Sonexa may:

- Pause playback
- Resume automatically
- Reduce playback volume
- Restore the original volume

Handling Audio Focus through Media3 simplifies implementation while ensuring consistent system behavior.

# Runtime Performance

Several implementation decisions contribute to Sonexa's runtime performance.

## Shared Playback Instance

Only one ExoPlayer instance exists throughout the application's lifecycle.

Every playback interface—including the Player Screen, MiniPlayer, Media Notification, Lyrics Screen, and future Android Auto support—communicates with the same player instance.

---

## Background Processing

Expensive operations execute asynchronously using Kotlin Coroutines.

Examples include:

- MediaStore scanning
- Room database operations
- Palette extraction
- Lyrics parsing
- Playback statistics
- Artwork loading

This prevents long-running tasks from blocking the main thread.

---

## Reactive Rendering

Compose observes immutable StateFlow objects throughout the application.

Whenever playback state changes, only the affected composables recompose, minimizing rendering overhead while maintaining smooth animations.

---

## Memory Management

Playback resources are allocated only when required and released during service destruction.

Examples include:

- ExoPlayer
- Audio Effects
- Bitmap resources
- Player listeners

Proper resource cleanup reduces memory usage and prevents hardware decoder leaks.
